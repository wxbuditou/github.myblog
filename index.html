springmvc学习

当我们协商Controller之后，就标记了它为spring的一个组件，并且是控制器的组件，此时我们的handlermapping会去扫描寻找这个controller是否与之匹配，如果发现撇皮就把这里处理的工作交给它。

匹配的规则又是什么呢

具体的匹配是通过请求的路径进行匹配的

@RequestMapping(URL)

此时就是通过这个URL进行匹配

RequestMapping解析

  • value 写的是路径，是一个数组的形式，可以匹配多个路径
  • path 是value的别名，二者任选其一，作用一样的
  • method 是可以指定可以访问的请求的类型，比如get，post
  • params 可以去指定参数，你还可以去限定指定某个值
  • headers 能够影响浏览器的行为
  • cousumers 消费者，媒体类型，可以限定必须为appication/json；charset=UTF-8
  • produces 产生相应的类型

关于请求路径的问题

  • ?任意的字符，斜杠列外
  • ‘*’ 0 到n，任意字符都可以，不能含斜杠
  • ** 支持任意层路径 /m3/** 这样才可以体现出来 /m3/a/b

GetMapping、PostMapping请求

  • getMapping限定了只支持get请求
  • postMapping限定了只支持post请求

  3. 可添加依赖设置成热部署compiler->bulid project automatically和shift+ctrl+alt+/ 点registry，勾选compiler.automake….，即可。

spring boot的版本仲裁中心：

以后我们导入依赖默认是不需要写版本：没有在dependencies里管理的依赖自然需要版本号

@SpringBootApplication

添加该注解，表示将类作为引导类。这个注解已经有多个注解的作用@Configuration，@EnableAutoConfiguration，@ComponentScan，通常会类前面加上MapperScan注解，指定需要扫描的包。启动该项目通过调用SpringApplication.run()方法，其中的参数为添加该注解的类。

在resources里面创建properties文件，可配置信息

在使用springboot的时候，发现异常:This application has no explicit mapping for /error, so you are seeing this as a fallback.

经过查询，将问题解决：

  1. Application启动类的位置不对.要将Application类放在最外侧,即包含所有子包
    原因:spring-boot会自动加载启动类所在包下及其子包下的所有组件.
  2. 在springboot的配置文件:application.yml或application.properties中关于视图解析器的配置问题:
    当pom文件下的spring-boot-starter-paren版本高时使用:
    spring.mvc.view.prefix/spring.mvc.view.suffix
    当pom文件下的spring-boot-starter-paren版本低时使用:
    spring.view.prefix/spring.view.suffix
  3. 控制器的URL路径书写问题
    @RequestMapping(“xxxxxxxxxxxxxx”)
    实际访问的路径与”xxx”不符合.
